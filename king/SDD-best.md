从 MixSpec 的设计与演进出发，回顾它如何从一个面向 spec-first、可落地 SDD 的工具，逐步走向 workflow、验证闭环、协作上下文与 Harness Engineer 的思路。核心观点是：在 AI Coding 时代，代码越来越便宜，真正有复利的是 spec、设计、约束、验证和沉淀。欢迎使用：https://docs.popo.netease.com/lingxi/3c56
这段时间一直在开发迭代MixSpec（好用可落地的SDD工具，自吹一波~，😁），我开始强烈地意识到一件事：

AI Coding 真正进入工程现场之后，决定效果差异的，越来越不是“模型会不会写代码”，而是“这套系统能不能把 AI 稳定地接进真实工作流”。

过去那句著名的话是：Talk is cheap. Show me the code.

这句话到今天依然成立，但它已经不足以概括当下的软件工程现实。

因为在 AI Coding 时代，代码本身正在快速变便宜。真正稀缺的，开始变成另外一些东西：需求能不能被结构化，方案能不能被约束，执行能不能被验证，结果能不能被沉淀成下一次的起点。

如果要把这表达，我更愿意写成：种变化压缩成一句更贴近今天的

Code is cheap. Show me the spec, the design, and the evidence.

这也是我重新理解 MixSpec 的过程。

如果一个人第一次看到 MixSpec，大概率不会先想到 agent、workspace 或多仓协作，而会先把它理解成一个 spec 工具，或者更宽一点说，一个 SDD 工具。它看上去天然会和 OpenSpec、SpecKit 这类东西放在一起比较。

而且，这种第一印象其实是对的。

因为如果回到它的起点，MixSpec 最开始强调的确实不是“多 agent 平台”，也不是“协作编排系统”，而是 spec-first。更准确地说，它最初想解决的是：如何把 SDD（Spec-Driven Development） 这件事，从“大家都觉得对”变成“一条真正能跑起来、能落地的工程路径”。

MixSpec 的演进过程，某种意义上也是一个工程视角转变的过程：从“怎么把 SDD 做实”，逐渐走到“怎么给 AI 和 Agent 设计一条可靠的 harness”。

OpenAI 在 2026 年 2 月那篇文章里，用一句话把这种分工说得很直接：Humans steer. Agents execute. 人类掌舵，智能体执行。这个表述对我很有启发，因为它把工程师的价值，从“亲手写每一行代码”，重新定义成了“设计环境、明确意图、构建反馈回路”。

所以这篇文章想讲清楚的，其实是三件事：

MixSpec 为什么会从一个 spec / SDD 工具出发

它为什么会一路演进到 workflow、验证、协作和 agent 问题

以及这个过程，为什么会自然通向 Harness Engineer 这套思路

一、MixSpec 的起点，是一个很朴素的问题：SDD 为什么一落地就会遇到很多现实摩擦？
如果你认同 spec 的价值，那你大概率也认同下面这些话：

先想清楚再写代码更好

先有 spec，再做 design，会减少返工

需求、方案、任务和实现之间，最好有一条清晰链路

这些道理几乎没人反对。真正的问题是，一进入真实开发现场，团队就很容易滑回另一种更常见、也更“顺手”的路径：

先基于现有产品文档或需求描述往下做

开发过程中一边实现，一边持续回改文档

临近联调、验收甚至上线时，还在补需求细节和边界条件

最后再靠 review、联调和测试兜底

这不是因为大家不相信 SDD，也不是因为 SDD 本身不能落地，而是因为它在真正落地时会持续遇到很多现实摩擦。它常常不够贴合真实开发习惯，也很难在团队里被长期建设和维护。一旦流程长一点、手工步骤多一点、规则需要靠人记忆去维持，团队就会自然退回到“先做起来、边做边补、最后集中兜底”的工作方式。

我当时真正想解决的，恰恰就是这个问题：不是再重复一遍“spec 很重要”，而是想把 spec-first 变成一条低摩擦、低心智负担、默认可执行的路径。



这也是为什么 MixSpec 早期做了很多看起来“内置很多”的设计。现在回头看，那些能力并不是为了显得系统更全，而是为了一个非常明确的目标：减少用户为了遵守 SDD 而额外承担的记忆成本和操作成本。

例如：

new 做成统一入口，不让用户自己判断该先跑什么

ff 负责 fast-forward，把明确需求直接推进到 spec + plan + tasks

constitution、模板、命令约定这些东西尽量内置，让项目约束自然出现，而不是靠团队手工维护

后续的 sync、drift、verify，逐步把一致性维护从“靠人自觉”变成“由系统托底”

所以如果今天要重新描述 MixSpec 的来源，我不会说它是把 OpenSpec、SpecKit、Compound Engineering “揉到一起”。这个说法太随意了，也会掩盖真正重要的设计目标。更准确的说法应该是：

MixSpec 是围绕“可落地 SDD”这个目标，对已有范式做了选择性吸收、重组和工程化落地。

起点不是方法论拼盘，而是工程落地。

但事情真正有意思的地方也在这里：当你真的试图把一个 spec 工具做成“可落地的 SDD 工具”时，你很快就会被迫去面对 workflow、验证、协作、上下文，最后自然走到 agent 和 harness 的问题上。换句话说，agent/workspace 不是 MixSpec 的第一眼定位，而是它沿着 spec-first 这条路继续演化后的结果。

二、从现在回头看，MixSpec 的演进，本质上是在给 SDD 补齐闭环
如果把 MixSpec 的迭代过程拉直，会看到一个非常连贯的方向：它从来不是突然转向“做 agent”，而是在一步步补齐 SDD 原本缺失的那几个环节。



 

1. 先把 change 立住，让开发有一个明确的工作单元
MixSpec 最核心的抽象之一，是 change。

每个 change 都对应一个目录，里面有固定的产物：

meta.yaml

spec.md

plan.md

tasks.md

这件事表面上看像是在整理目录，实际上解决的是一个更基础的问题：如何把一次开发动作，从松散的对话和零散的修改里抽出来，变成一个有边界、有上下文、有生命周期的单元。

对人来说，这解决的是叙事问题：这次改动到底是为了解决什么，做到哪一步了，接下来该往哪走。

对 AI 来说，这解决的是上下文问题：你到底是在为哪个 change 工作，你的输入和输出应该落在什么地方。

如果没有这一层，后面的 spec、plan、tasks，都会变成漂浮在聊天记录上的临时产物。

2. 再把 spec、plan、tasks 做成一条产物链，而不是一堆文档
有了 change 之后，下一步就是把 SDD 真正站稳。

这里最重要的不是“多写几份文档”，而是让不同阶段的产物变成一个稳定的输入输出协议：

spec 负责沉淀业务意图

plan 负责表达技术约束和实现方向

tasks 负责拆成可执行任务



这一步的关键，在于把“对话”变成“结构化产物”。

聊天当然方便，但聊天有几个天然问题：它不可 diff，不容易 review，很难在多轮上下文里保持稳定，也不适合作为多个 agent 之间传递的协议。spec / plan / tasks 的价值，不只是更规范，而是它们让上一个阶段的产物，天然成为下一个阶段的输入。

这时的 MixSpec，仍然是在做 SDD，但它已经不是一个“辅助写文档”的工具，而是在慢慢形成一条由产物驱动的执行链。

3. 再往前走，就必须处理一致性问题
当一条 spec -> plan -> tasks -> implementation 的链真的开始运转，新的问题就马上出现了：这条链非常容易过期。

典型情况几乎每个团队都见过：

spec 改了，plan 还是旧的

plan 调整了，tasks 没有同步

代码已经变了，spec 却没有更新

已归档的文档被手工修过，没人知道基线已经漂移

MixSpec 后来的很多关键能力，几乎都在围绕这个问题展开：

sync 负责 spec -> plan -> tasks 的级联同步

drift 负责归档后漂移扫描

verify 把 lint、type、tests、tasks 完成度、spec-truth 风险拉到同一个检查点

archive 不只是归档文件，同时顺手沉淀 learning

这一层非常关键。因为一旦走到这里，MixSpec 关注的就不再只是“文档有没有写”，而是“这条文档链能不能持续保持和实现链一致”。

也就是说，它开始从“规范化工具”走向“工作流基础设施”。

三、也正是在这里，我开始理解 Harness Engineer
当 AI 还只是一个聊天助手时，很多问题看上去像是 prompt 问题。

但一旦 AI 开始真正进入工程链路，你会发现事情完全不是这么简单。真正麻烦的问题会变成：

Agent 应该在哪一步介入

它要读哪些产物

它的输出如何被下游继续消费

它犯错时，系统怎么发现、怎么阻断、怎么回退

最后又该如何把这次执行变成可复用的经验

这时，工程的重心就会从“生成内容”转向“设计轨道”。

这也是我理解的 Harness Engineer 的核心：它不是一个“更会写 prompt 的人”，而是一个专门负责为模型和 agent 设计工作轨道的人。

这个轨道至少要解决四件事：

给定边界

提供上下文

设定约束

建立验证与回收机制

可以把它理解得更直观一点：模型像一匹马，prompt 只是口头指令，context 是它能看见的路，而 harness 是整套“鞍、缰绳、脚蹬、路线图和回拉机制”。它不只是告诉模型“做什么”，还决定模型能不能稳定地做完、做对、做得可复用。

所以 Harness Engineering 也不是 Prompt Engineering 的简单升级版。它至少还包括：

tools：模型能调用什么工具、这些工具是否稳定

files：哪些产物被版本化保存在仓库里

hooks：在哪些阶段强制检查、阻断或回退

memory：什么信息可以跨任务、跨会话复用

verification loops：系统有没有把“做完”改造成“验证完才算做完”

如果用这个视角回头看 MixSpec，会发现它的很多设计其实都在做同一件事：

change 在定义工作边界

spec / plan / tasks 在定义阶段产物

sync / drift / verify 在定义纠偏和门禁

archive / learnings 在定义沉淀和复用

它们合在一起，已经不只是一个命令集合，而是一套逐渐成形的 harness。

四、MixSpec 真正的变化，不是命令越来越多，而是系统重心在变化
如果只盯着功能列表看，你会觉得 MixSpec 只是越来越“全”。但如果从设计重心看，它的变化其实很清楚，而且非常有层次。

第一阶段：把 spec-first 变成一条能跑起来的默认路径
早期最核心的事，是让用户愿意走这条链路。

所以重点一直是：

降低进入门槛

降低流程记忆成本

通过内置模板、命令和约定，把 SDD 从理念变成默认操作

这对应的是“把 SDD 做实”。

第二阶段：把文档链和实现链闭环
后来重点变成了治理一致性：

文档改了，怎么同步

实现变了，怎么发现漂移

归档之后，怎么保住真理源

学到的经验，怎么回到下一次 change

这对应的是“把执行闭环做实”。

第三阶段：让多个 agent、多个 repo 仍然围绕同一个 truth 工作
最近关于 epic、lineage、link、workspace、agent team 的思考，又把问题推进了一层：

多个 repo 如何共享上下文

接口契约该在哪一层锁定

多个 agent 如何在 plan 阶段就完成对齐，而不是等到 apply 或联调时才暴露冲突

这时候，问题已经不再只是“文档怎么写”，而是一个更典型的 harness 问题：如何定义 agent 的工作边界、共享上下文和协作协议。

这里我觉得 OpenAI 那篇文章还有两个判断特别值得借来理解 MixSpec。

第一，the goal is agent legibility。目标不是单纯让系统“对人类可读”，而是让智能体也能直接读懂。很多知识如果还停留在聊天记录、脑内记忆、临时文档里，对 agent 来说就等于不存在。

第二，give Codex a map, not a 1,000-page manual。不是把所有规则堆进一个巨大的 AGENTS.md，而是给它一个稳定入口、一套分层索引、一组可按需展开的产物。

这两点和 MixSpec 的方向其实很贴：

change 目录是在给 agent 一个稳定入口

spec / plan / tasks 是在把上下文变成按阶段展开的地图

constitution、config、learnings 是把隐性知识从人脑和对话里迁进仓库

workspace 则是在多 repo 条件下继续维持这种“可读地图”


 

所以今天回头看，MixSpec 的演进并不是“先做了个 SDD 工具，后来又顺手做点 agent 功能”。更准确地说，它是沿着 practical SDD 这条路一路往前走，走着走着自然进入了 harness 的问题域。

五、为什么我不再满足于“show me the code”
在真正用 AI 参与开发之后，我有一个很明显的心态变化。

一开始我也会本能地问：AI 到底能帮我写多少代码？

但一旦你真的跑过几轮 ff、apply、verify，就会意识到：能写出代码，已经不是最难的事了。

真正难的是：

这段代码是不是沿着正确的问题定义长出来的

它是不是在清晰约束之下生成的

有没有足够证据证明它值得被信任

下一次类似任务，团队能不能在这次结果之上复用，而不是重新开始

所以今天如果我要判断一个 AI 工程体系成熟不成熟，我不会只看它能写多少代码，而会看它有没有这些东西：

稳定的 spec 和 plan 结构

清晰的执行链路

可追踪的验证证据

可复用的 learnings

明确的 agent 工作边界

这也是为什么我更愿意把那句话改写成：

Code is cheap. Show me the spec, the design, and the evidence.

代码仍然重要，但它越来越像结果，而不再是能力的全部。真正决定系统质量的，是代码背后的设计思路、产物结构和验证链路。

六、接下来，MixSpec 更像是在长成一套 Harness 基础设施
沿着现在的规划往前看，MixSpec 后面的重点其实很一致。

Workspace：给 agent 正确的地形，而不是控制台
Workspace 要解决的，不是“我能不能发明一个新的 agent 调度框架”，而是一个更根本的问题：

在 agent 的眼里，这个项目世界应该长什么样？

它关心的是：

工作区里有哪些 repo

当前 active change 是什么

哪些上下文是共享的

哪些契约应该在 plan 阶段就锁定

也就是说，workspace 的重点并不是“控制 agent”，而是“给 agent 正确的地形”。

如果沿着 OpenAI 的说法，它其实是在继续把仓库变成 system of record。也就是：让真正决定 agent 行为的知识，尽量存在仓库里、被版本化、能被检索、能被校验，而不是散落在消息、会议和个人经验里。

Agent Team：重点不是多开几个 agent，而是定义清楚分工和接口
Agent Team 也一样。

真正有价值的，不是把多个 agent 同时拉起来，而是把下面这些问题定义清楚：

哪类 agent 负责 spec

哪类 agent 负责 plan

哪类 agent 负责 implementation

哪类 agent 负责 review 和 QA

从这个角度说，MixSpec 在这里真正提供的，也不是人格设定，而是产物协议和流程边界。

测试链路打通：让“验证”成为一等产物
最后一个我觉得会越来越关键的方向，是测试链路的继续打通。

现在 MixSpec 已经把 lint、type、tests、tasks 完成度、spec-truth 风险、archived drift 收进了一条统一检查链。但如果继续往前走，真正重要的是把“验证证据”提升到更核心的位置。

未来值得继续补的能力包括：

在 plan 里显式约束验证策略和测试范围

在 tasks 里明确产出测试用例、回归 case 或自动化脚本

自动触发或统一汇总构建、单测、集成测试、端到端测试结果

跨 repo 的接口契约校验

更明确的回归测试链路

UI 截图、录屏、联调结果等多模态 evidence

change 级别的验证汇总

走到这一步，MixSpec 才会更完整地体现出 Harness Engineer 的思路：不是只让 agent 会写，而是让 agent 的输出可验证、可比较、可复盘。

七、结语：MixSpec 的起点是 practical SDD，但它自然会走向 harness
如果要用一句话概括我现在对 MixSpec 的理解，大概会是这样：

它最开始是在做 spec-first，在做可落地 SDD；但当这条链真正开始被 AI 执行之后，它自然就走向了 harness 的问题。

所以今天我更愿意这样描述它：

它的起点，是 practical SDD

它中途补齐的，是执行闭环和一致性治理

它接下来要长成的，是多 agent、多 repo 条件下的 harness 基础设施

如果再压缩一点，就是：

模型会越来越可替换，代码会越来越便宜；真正会产生复利的，是你如何组织 spec、设计、约束、验证和沉淀。

而这，恰恰就是 Harness Engineer 要做的事。

从这个意义上说，我现在更愿意把 MixSpec 看成这样一个东西：

它表面上是在做 spec、plan、tasks、verify；但更深一层，它是在试着回答一个越来越重要的问题：

当“人类掌舵，智能体执行”开始成为真实的软件工程组织方式时，我们该如何把意图、上下文、约束和证据，稳定地装进一套可持续演化的 harness 里？

如果说过去软件工程的核心资产是代码，那么在 AI 时代，核心资产会越来越像是另外一组东西：结构化的 spec、可执行的 workflow、明确的约束、可靠的验证链，以及能不断复用的工程记忆。

代码当然仍然重要。但它越来越像结果，而不是全部。

MixSpec 对我来说，正是在沿着这条方向演化：从一个 spec / SDD 工具出发，慢慢长成一套让 AI 真正能进入工程现场的 harness。
