## 1. 实验口径

*   服务：`qwen3-5-classifier`，复用 `5` 号卡上已启动的 `/classify` 服务
*   样本来源：`qwen35_classifier_4090d_20260331` 对应压测目录中的原始大盘随机抽样 \*&#x20;
*   并发档位：`1 / 2 / 4 / 8 / 16`，每档 `2000` 张图&#x20;
*   图片口径：接口侧先压到 `<= 1.0MB`，计费 token 使用 `ceil(min(width*height, 349525) / 1764)`&#x20;
*   容量约束：只接受 `P99 < 1000ms` 且 `error_count = 0` 的并发档位

## 3. 单卡容量结论

*   满足 `P99 < 1s` 且零错误的最高并发为 `1`，单卡实测吞吐为 `2.77` QPS。

## 6. 结论

### 6.1 容量基线

| 指标     | 数值            | 说明             |
| ------ | ------------- | -------------- |
| 单卡 QPS | 2.77          | 满足 P99 < 1s 约束 |
| 单卡日处理量 | 42.22 Mtoken  | 满负载 24h        |
| 单卡理论成本 | 0.51 元/Mtoken | 满负载最优值         |

### 6.2 部署建议

*   **最小可行配置**：`11` 卡 / `2` 台机器，支持 `30.47` QPS，满足峰值需求
*   **推荐冗余配置**：`12` 卡，冗余 `10.8%`，月成本增加 `646.50` RMB，提升故障容错能力

### 6.3 成本与盈亏分析（基于 11 卡配置）

> 以下分析基于 `11` 卡部署（日成本 `237.05` RMB），按 `30 QPS` 峰值、`176.40` token/请求计算

### 6.4 关键结论

1.  **盈亏平衡点**：日均峰值运行 `≥ 6h` 方可覆盖机器成本
2.  **成本敏感性**：峰值利用率每提升 `1h`，单位成本下降约 `0.26` 元/Mtoken
3.  **定价建议**：若业务峰值集中在 `4h` 以内，需将定价上调至 `≥ 3.2` 元/Mtoken 才能保本
